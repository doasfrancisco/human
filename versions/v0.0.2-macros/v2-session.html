2. v0.0.1 — TLA+ Syntax Compiler

Input (TLA+) 

---- MODULE HelloWorld ----
VARIABLE printed
Init == printed = FALSE
Print == printed = FALSE /\ printed' = TRUE
Next == Print
TypeInvariant == printed \in {TRUE, FALSE}
Spec == Init /\ [][Next]_printed
====

Output (Python) 

import random

class HelloWorld:
    def __init__(self):
        self.printed = False

    def print(self):  # TLA+: Print
        if self.printed == False:
            self.printed = True
            return True
        return False

    def step(self):
        actions = []
        if self.printed == False:
            actions.append(self.print)
        if actions:
            random.choice(actions)()

    def check(self):
        assert self.printed in {True, False}, "TypeInvariant violated"

3. Key Realizations

TLA+ looks like Python for simple things

User: "I see the code in TLA+ for fizzbuzz and it's similar to a simple python. Is there a way to make it look similar to English?"

State machines aren't always the right abstraction

1 + 3 + 5 doesn't need state machine decomposition. Pure expressions are already the most compressed form. State machines earn their keep when things change over time, have rules, or involve concurrency.

Compression = separating the idea from the machinery

FizzBuzz is not an if/elif chain. It's a rule: {3: "Fizz", 5: "Buzz"} . The if/elif chain is machinery that implements the rule. 

Good compression: divisors {3: "Fizz", 5: "Buzz"}
Bad compression: same logic with fewer characters

LLMs are part of the language

Pipeline: English → (LLM) → fran++ → (compiler) → Python

User says "fizzbuzz" → LLM knows the algorithm → emits compressed form → compiler produces Python. Without the LLM, fran++ is just another syntax.

Rules and goals, not steps

From studying real TLA+ specs (DieHard, CoffeeCan, GameOfLife via Nia-indexed repos):

"TLA+ is most powerful NOT when it describes an algorithm step by step. It's powerful when it describes rules and goals and lets the machine figure out the steps. "

Two kinds of problems

Type Example You provide
Transformations FizzBuzz, sorting, 1+3+5 The steps (you know them)
Search problems DieHard, puzzles, scheduling Rules + goal (machine finds steps)

The spec (water_jugs.fpp) 

module WaterJugs

variables
    big = 0
    small = 0

action FillBig
    set big to 5

action FillSmall
    set small to 3

action EmptyBig
    set big to 0

action EmptySmall
    set small to 0

action PourSmallToBig
    set big to min(big + small, 5)
    set small to small - (min(big + small, 5) - big)

action PourBigToSmall
    set small to min(big + small, 3)
    set big to big - (min(big + small, 3) - small)

constraint
    big between 0 and 5
    small between 0 and 3

goal
    big = 4

Compiled Python (water_jugs.py) 

from collections import deque

def solve():
    initial = {"big": 0, "small": 0}

    def fill_big(s):
        return {**s, "big": 5}

    def fill_small(s):
        return {**s, "small": 3}

    def empty_big(s):
        return {**s, "big": 0}

    def empty_small(s):
        return {**s, "small": 0}

    def pour_small_to_big(s):
        return {**s, "big": min(s["big"] + s["small"], 5),
                "small": s["small"] - (min(s["big"] + s["small"], 5) - s["big"])}

    def pour_big_to_small(s):
        return {**s, "small": min(s["big"] + s["small"], 3),
                "big": s["big"] - (min(s["big"] + s["small"], 3) - s["small"])}

    all_actions = [
        ("FillBig", fill_big), ("FillSmall", fill_small),
        ("EmptyBig", empty_big), ("EmptySmall", empty_small),
        ("PourSmallToBig", pour_small_to_big),
        ("PourBigToSmall", pour_big_to_small),
    ]

    def is_valid(s):
        return 0 <= s["big"] <= 5 and 0 <= s["small"] <= 3

    def is_goal(s):
        return s["big"] == 4

    # BFS explores all states, finds shortest path
    queue = deque([(initial, [])])
    visited = {tuple(sorted(initial.items()))}

    while queue:
        state, path = queue.popleft()
        if is_goal(state):
            # print solution...
            return
        for action_name, action_fn in all_actions:
            next_state = action_fn(state)
            key = tuple(sorted(next_state.items()))
            if key not in visited and is_valid(next_state):
                visited.add(key)
                queue.append((next_state, path + [(action_name, next_state)]))

solve()

Result 

Solution found in 6 steps:

  Start: big=0, small=0
  FillBig -> big=5, small=0
  PourBigToSmall -> big=2, small=3
  EmptySmall -> big=2, small=0
  PourBigToSmall -> big=0, small=2
  FillBig -> big=5, small=2
  PourBigToSmall -> big=4, small=3

32 lines of English → 66 lines of Python BFS solver → shortest solution found automatically.

You described the puzzle. The machine solved it.

6. Open Questions
